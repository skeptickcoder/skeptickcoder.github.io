Case study · JobFinder

By the time a job hits LinkedIn, it's days old with hundreds of applicants. I wanted to be upstream of that.

So I built JobFinder: it reads the applicant-tracking APIs behind 53 careers pages directly, scores every new posting with a local LLM against my profile, and delivers a ranked digest to Telegram — twice a day, before the crowd arrives.

The insight

    
Careers pages are frontends. Read the API instead.

    

      Almost every careers page is a thin frontend over an applicant-tracking system — Greenhouse,
      Lever, Ashby, Workday — each with a public JSON API designed for exactly this kind of read.
      Going to the source means structured postings the moment they publish: no scraping, no HTML
      parsing, no bot detection — just one small adapter per ATS.
      Seven adapters now cover 53 companies plus two VC-portfolio network boards,
      about 4,300 postings per scan.

dual scoring

          
A fast keyword score against my skills profile, then a local LLM (qwen2.5:14b via Ollama) rates each new match 1–10 with reasoning and red flags. Free, private, provider-swappable.

Under the hood

    
Three details engineers ask about

    


      

        
          +
          
Delivery-safe state

          at-least-once, by design
        
        

          
seen_jobs.json is written only after deliver_report() returns.
          If Telegram is down, the process crashes, or the network drops mid-send, nothing is marked seen —
          every job re-reports on the next scan. That's a deliberate at-least-once choice:
          in a job search, a duplicate notification costs three seconds; a silently dropped
          posting can cost the opportunity.

        

      


      

        
          +
          
Company resolver with collision handling

          "Orca" could be three companies
        
        

          
Adding a company takes one pasted careers URL. The scanner probes Greenhouse, Lever, and
          Ashby by slug, validates the token against the live board before trusting it, and caches
          the result (failures too, with a 3-day retry TTL). Validation matters because names collide —
          "Orca" alone could be three different companies on three different ATSs.

          
The same discipline applies across sources: when a job appears both on a company's own board
          and on a VC-network aggregator, the direct listing wins (matched on normalized
          company + title), and an aggregator's careers-page copy beats its LinkedIn copy. One job,
          one notification.

        

      


      

        
          +
          
HTTP-native caching

          unchanged boards cost 0 bytes
        
        

          
Every board is revalidated with ETag / If-Modified-Since; an unchanged
          board answers 304 and transfers nothing. The cache key includes a hash of the config,
          so editing a keyword forces a clean refetch. Greenhouse gets an extra trick: fetch the light
          job list first, then one ~12 KB detail request per matching job only.

The skeptic's corner

    
Questions worth asking me

    

      

        Why not just scrape LinkedIn?
        
LinkedIn is downstream — postings arrive there days late, and scraping it means fighting
        bot detection while violating its terms. The ATS is the origin: first-party, structured
        JSON, published the moment the role opens. Upstream beats faster-downstream.

      

      

        Is calling these APIs actually allowed?
        
They're public, unauthenticated endpoints built for careers-page embedding — the same calls
        your browser makes when you open the page. JobFinder is deliberately polite about it: one light
        request per company per scan (or a 0-byte 304), 1-second per-host spacing, backoff on 429,
        an honest User-Agent, twice a day. A scan is less load than one human visitor.

      

      

        Why a local LLM instead of a hosted API?
        
Job-search data is personal, and at this scale (~5 s per new match on an M-series Mac)
        a local qwen2.5:14b via Ollama is free, private, and plenty accurate — it correctly buries an
        internal-facing "Product Manager" at 2/10 while ranking a Solutions Engineer role 9/10.
        The provider is a config switch: Anthropic-backed scoring with structured output is already
        implemented for when scale or quality demands it.

      

      

        What breaks if Telegram is down?
        
Nothing is lost. State saves only after delivery succeeds, so an outage means the same jobs
        simply re-report on the next scheduled scan. The failure mode is a duplicate message,
        never a missed posting — chosen deliberately, because the two costs aren't symmetric.
